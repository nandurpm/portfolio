Personal Notes on Learning New Technologies

February 25, 20267 min readPersonal Writings

Learning new technology is easy to start and difficult to sustain. Tutorials, tools, frameworks, and AI-generated examples can create the feeling of rapid progress, but real capability appears only when the learner can solve a problem without depending on the exact demonstration they copied.

I try to learn with a practical output in mind. Instead of saying “I want to learn JavaScript,” I define a result such as “I want to build a searchable project list,” “I want to automate a repeated Excel task,” or “I want to read sensor values and display them clearly.” The output gives direction and reveals which knowledge is actually missing.

Start from the problem, not the trend

A popular tool is not automatically useful for every person. Before choosing what to learn, I ask whether it supports my work, solves a repeated problem, strengthens a core skill, or enables a project I genuinely want to complete. This prevents learning from becoming a collection of disconnected introductions.

Build a foundation before chasing speed

Shortcuts are useful only when the fundamentals are understood. In electronics, voltage, current, grounding, measurement, and signal paths remain important even when the controller changes. In web development, HTML structure, CSS layout, browser behaviour, and JavaScript logic remain important even when a framework changes.

Fundamentals reduce dependency on a specific tutorial. They also make errors easier to diagnose because the learner can reason about the system rather than trying random fixes.

Use small projects to create feedback

A small finished project teaches more than a large unfinished plan. The project should be slightly beyond current ability but limited enough to complete. A useful sequence is:

  1. Define one clear output and the minimum features.
  2. Build the simplest working version.
  3. Test it with realistic input and failure cases.
  4. Document what was difficult and what was changed.
  5. Improve structure, usability, and reliability after it works.

The first version does not need to be impressive. Its purpose is to expose gaps in understanding. The improvement stage develops engineering judgement.

Separate using a tool from understanding it

Modern tools can generate code, formulas, layouts, and explanations quickly. They are valuable, but copied output can hide missing knowledge. I consider a task learned only when I can explain the main logic, change the behaviour, identify common failure points, and verify the result.

A tool should increase the speed of thinking and execution, not replace the responsibility to understand and verify the output.

Keep notes that are useful later

Learning notes should capture decisions, commands, formulas, error causes, and working examples. Copying an entire article into a notebook is less useful than recording the exact issue encountered and the correction that solved it.

I prefer notes with a clear title, short explanation, tested example, expected result, and warning about common mistakes. Over time, these notes become a personal reference library and reduce repeated searching.

Protect time and attention

Daily learning does not require several uninterrupted hours. Consistent focused sessions are more realistic. A 30-minute session can be enough to complete one test, understand one concept, or document one result. The key is to decide the task before starting.

Constantly switching between tools creates shallow familiarity. I try to keep one primary learning project active and place other interesting topics in a later list. Finishing creates confidence and evidence; endless exploration creates clutter.

Review progress through outputs

Certificates and completed videos are easy to count, but they do not prove independent ability. Better evidence includes a working file, published page, tested circuit, clear drawing, automated workflow, or written explanation. These outputs also become portfolio material when they are complete and honestly presented.

Accept that relearning is normal

Knowledge that is not used will fade. That is not failure. A documented project makes relearning faster because the reasoning and working example remain available. The goal is not to memorise every command. The goal is to understand the structure well enough to recover quickly and continue.

Long-term learning is less about motivation and more about system design: choose useful problems, keep the scope controlled, practise fundamentals, verify outputs, document lessons, and repeat. Small completed cycles create more professional growth than occasional bursts of ambitious but unfinished study.
